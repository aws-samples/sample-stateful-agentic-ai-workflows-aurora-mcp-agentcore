Chapter
I— What we’re building

One sentence, then the rest.

A traveler describes a trip in their own words; Meridian plans, books, and watches it.

That’s the whole product, in one sentence. The rest of this document is just unpacking what each verb means, and what infrastructure makes them possible.

“A slow week somewhere we can drink good wine.” The kind of input Meridian is built for

The current code already proves the bones: Aurora PostgreSQL for retrieval and memory; the Strands Agents SDK for orchestration; MCP for tool access; Cohere Embed v4 for semantic search. The chapters that follow show how the experience climbs on top of that.

Chapter
II— The four phases

A teaching ladder, not a toggle.

Each phase adds exactly one capability — so the same query can be re-played up the ladder, and the audience can see where each piece earns its keep.

  1. Phase 1 · Filters. Plain SQL on trip_packages via the RDS Data API. The fastest possible answer for an exact match.
  2. Phase 2 · MCP. Same queries reframed as MCP tool calls. Tools become the contract.
  3. Phase 3 · Intent. Hybrid pgvector + tsvector. “Romantic week in Italy” resolves to packages.
  4. Phase 4 · Memory. The supervisor grounds every turn in traveler_preferences recalled by a Strands @tool memory agent.

In the workshop, the audience scrubs through 1 → 4 on the same query and watches the answer improve. The scaling story (50 → 50K trips/day) sits underneath each rung.

Chapter
III— Memory of the traveler

Every fact, with a source.

Memory is the difference between a chatbot and a concierge. Meridian stores it in Aurora, auditable, editable, and forgettable.

A dedicated memory agent writes facts; the supervisor only reads. Each row carries a confidence score, the conversation that taught it, and the timestamp. Recall is semantic via pgvector and exact via b-tree on (traveler_id, key).

Chapter
IV— Trace as a product

The thing devs want to copy.

The chat is the user surface. The trace is the developer surface. Both come from the same source of truth — a row in agent_traces with a JSONB span tree.

Every turn produces a Gantt-style timeline: orchestration → memory → tools → model → synthesis, with retries and confirmation gates inline. Spans are OpenTelemetry-compatible; replay re-runs against cached tool responses.

For the traveler

The itinerary view

Days, events, statuses (held / confirmed / disrupted). Same trace, friendly skin.

For the developer

The Gantt view

Spans, latencies, SQL, token spend, cache hits. Permalink-able and replay-able.

Chapter
V— The substrate

Aurora plus MCP, made legible.

Two surfaces every reviewer asks for: the schema and the tool catalog. Both are first-class pages in the app, not docs-folder afterthoughts.

Schema, in one paragraph

trip_packages holds the catalog with a vector(1024) embedding and a generated tsvector. travelers, traveler_preferences, and trip_interactions hold identity and memory. conversations / conversation_messages hold session state. bookings / booking_lines hold reservations. agent_traces holds JSONB span trees per turn. Nine tables; everything else derives.

MCP, in two sentences

Every external system — Aurora reads, traveler memory, weather, payments, the GDS — sits behind an MCP server with a versioned schema. The Tool Catalog lists each one with its version, p50 latency, owner, and a dry-run.

Chapter
VI— Governed action

Plan, confirm, commit.

Before any irreversible step — charging the card, locking a non-refundable rate — the agent presents the plan with a diff. The user confirms once. The receipt links back to the trace.

  1. Plan. Supervisor proposes the smallest set of actions that satisfies the request inside the user’s budget.
  2. Confirm. The diff shows what changes — totals, refundable rates, scope.
  3. Commit. One click; the receipt links to the trace and the audit row in bookings.

Per-agent budgets, scope rules, and refund caps are enforced in the booking agent — not in the prompt. The leash is visible, and verifiable.

Chapter
VII— Long-running trips

Wakes the agent on disruptions.

A trip lives for weeks. Aurora persists every step as a state machine; a worker wakes the supervisor on triggers (gate change, weather flag, price drop), and the user is told only the things that need attention.

The agent acts silently within scope, and asks only when it can’t. The default mode for any trip Meridian is watching.

Disruption budgets and re-book scopes are first-class fields in bookings. Logical replication streams change events to the UI over SSE so the itinerary view updates live without polling.

Chapter
VIII— What ships first

The smallest bet that already tells the story.

Two waves. Each wave is small enough to scope to a single sprint, and large enough to change what the demo feels like.

WaveWhat shipsWhy now
01 Trace permalinks · Memory inspector Highest narrative payoff for least new infra.
02 Supervisor + 4 specialists · MCP catalog Turns “Phase 4” into something teams can own.
03 Plan/confirm/commit · long-running ops Production-shape governance and durability.

Beyond Wave 03, the open questions are operational: replay sandbox, live ops dashboard, multi-traveler memory. They’re scoped but they’re not load-bearing for the story.
